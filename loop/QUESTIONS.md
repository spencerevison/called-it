- T25: no decision-detail page exists yet (T27), and the edit page 404s once a decision leaves `draft` (T21/T22), so the pre-mortem panel only has one place to render right now — the edit page — and is only ever exercised in the draft state. `add-own-risk`/`regenerate` are still gated in the server actions themselves (draft-only, independent of the UI), so once T27 adds a read view for active decisions the same panel can render there with `isDraft={false}` and the buttons will already be hidden/no-op correctly. Ownership for `addUserRisk` is checked via the premortem row (`premortem.user_id`), not by re-deriving it through the decision, per the historical IDOR fix noted in docs/EVIDENCE.md (a premortem_id doesn't imply the caller's decision). Also: `premortems.decision_id` has no uniqueness constraint, so "regenerate" inserts a brand-new premortem row each time rather than updating in place — the edit page fetches only the most recent premortem (`order by created_at desc limit 1`) and treats earlier ones as inert history; not stated explicitly in DATA_MODEL, smallest-reasonable read of "regenerate allowed." **Update (review fix):** the "new row per regenerate" design initially let a user's own `source=user` risks get silently stranded on the now-inert old row — `generatePremortem` now looks up the previous latest premortem before inserting the new one and copies its `source=user` rows forward onto the new premortem (same compensating-delete-on-failure pattern as the AI-risk insert). `addUserRisk` also now re-checks that its `premortemId` is still the decision's latest premortem before inserting, closing a stale-tab write-to-nowhere case where a regenerate in another tab left the form pointed at a dead row. Both are still outside T26's not-yet-built transaction, so the same TOCTOU note below extends to risk inserts, not just premortem inserts.
- T26: commit is implemented as a single Postgres function (`commit_decision`, migration `20260703000001`) called via `service.rpc(...)` rather than a sequence of app-layer inserts with compensating deletes — a real DB transaction is the only way to make "partial failure leaves draft untouched" true by construction instead of by careful cleanup ordering (the T24 review found exactly this class of bug with the compensating-delete approach). The function takes `p_user_id` as a plain parameter rather than deriving it from `auth.uid()`, so `execute` is revoked from `anon`/`authenticated` and granted only to `service_role` — otherwise any signed-in caller could invoke the RPC directly with someone else's user id (bypassing the server action's `auth.getUser()` check) and commit a decision they don't own. Row is locked `for update` inside the function so a concurrent commit sees the post-commit status before writing — this addresses T24/T25's carry-forward note for the commit side of the TOCTOU (premortem-generation's own read is still an unlocked SELECT; fully closing that would mean touching T24/T25's files, out of scope for T26). No decision-detail page exists yet (T27), so a successful commit redirects to `/` — same "nothing to redirect to yet" gap noted for T22.

- T28: `decision_event_type` has no status-changing member for revise/reverse/reaffirm — `decisions.status` stays `draft|active|resolved|abandoned` per DATA_MODEL, matching M8's definition of "reversed" as an event on an otherwise-still-active decision, not a status. All three actions are gated to non-draft decisions only (SPEC F1: "decision becomes append-only except via logged events" once committed); a draft has no committed baseline to revise/reverse/reaffirm against. `revise` and `reverse` both store their text in the generic `payload.note` field per DATA_MODEL's own comment (`payload jsonb -- e.g. {note}`) — reverse enforces the note as required per the AC ("prompts for a one-line reason"), revise also requires one (a revision with no description of what changed isn't useful for the timeline), reaffirm takes no payload at all.
