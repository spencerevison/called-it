- T03: SPEC doesn't name exact env var names for Trigger.dev/Langfuse/Resend, so `.env.example` uses each service's own SDK-conventional names (`TRIGGER_SECRET_KEY`/`TRIGGER_PROJECT_ID`, `LANGFUSE_SECRET_KEY`/`LANGFUSE_PUBLIC_KEY`/`LANGFUSE_BASE_URL`, `RESEND_API_KEY`). Small addition beyond the literal index list, not a schema-shape decision.
- T11: local Supabase's `api.auto_expose_new_tables` now defaults to false (matches the new cloud default — confirmed in `supabase/config.toml`'s commented-out `# auto_expose_new_tables = true`), so newly created tables get no GRANTs to `anon`/`authenticated`/`service_role` through PostgREST even with RLS policies in place. Added standard-Supabase-convention broad GRANTs (+ `alter default privileges`) to the T11 migration so the RLS policies are actually reachable/exercisable through the API layer — not spoken to explicitly in DATA_MODEL.md, but required for policies to mean anything through PostgREST (T05–T10 never hit this since they verified via direct `psql`).
- T11: added `src/lib/supabase/service.ts` (service-role client) ahead of schedule — T03's note deferred it to "the first server action that needs service-role writes," but the T11 policy tests need a service-role client to seed/clean fixtures that bypass RLS. Minimal wrapper, same shape as the existing browser/server helpers.
- T11: the task's AC also names "the server-action layer rejects a child row whose parent belongs to another user or another decision" — no server-action layer exists yet (DATA_MODEL rule 4, cross-decision `linked_risk_id` lineage), so that assertion isn't testable at T11. - T13: METRICS.md says "a separate aggregation service maps DB rows → these inputs" — read M6 (short/long horizon partition) and M9 (valence-from-forecasts derivation) as T20's job, not T13's; `horizonCalibrationGap`/`selfServingIndex` take pre-partitioned/pre-derived inputs rather than raw horizon-days or forecast lists. M10's vector gives only aggregates (5 decisions/8 knowable failures/5 linked/4 decisions with a link), not a decision-by-failure breakdown, so the test constructs one specific grouping that reproduces those aggregates — the exact assignment isn't normative, only the aggregate numbers are.
- T21: task text says "protected app layout" but no route-group scaffolding exists yet for the not-yet-built decision routes (T22+), so protection is enforced at `src/middleware.ts` (allowlisting only `/login` and `/auth/confirm`) rather than a dedicated `(app)` route group; the root `layout.tsx` is auth-aware only for the sign-out control. Whichever task first adds a route group should keep the middleware as the actual enforcement point — it already covers every current and future path by default (deny-by-default beats an allowlist of protected routes). Login origin is derived from the `host`/`x-forwarded-proto` request headers rather than a `NEXT_PUBLIC_SITE_URL` env var (SPEC doesn't define one) — matches the Supabase docs' own example for `emailRedirectTo`.
- T22: no form-validation library exists (zod/react-hook-form not installed) — implemented inline field parsing/validation in `src/app/decisions/actions.ts` server actions rather than adding a dependency; DB writes go through the service-role client (per T11's established pattern: RLS on `decisions` is select-own only, no insert/update policies), with ownership + draft-status checks done in the action. `updateDecision` returns the same "Decision not found." error for a nonexistent row and for a row owned by another user, to avoid confirming another user's decision id exists (not stated in the AC, but T25's EVIDENCE.md IDOR note made this the obvious default). No decision list/detail page exists yet (T27), so a successful create redirects to the new decision's own `/decisions/[id]/edit` rather than a list — smallest reasonable landing spot given current scope.
- T24: `premortems.prompt_version` has an FK to `prompt_versions(id)` (added in T10), but `prompt_versions` is only populated at T37 (prompt registry parses `prompts/*.md` and inserts rows). The live pre-mortem path (gated on `ANTHROPIC_API_KEY`, exercised only manually before T37 lands) will hit an FK violation inserting `prompt_version: "premortem_v1"` until T37 runs; unit tests mock the service client so this doesn't surface there. Same task-ordering gap the DATA_MODEL comments already accept for T08→T10's FK.
- T24: added `@anthropic-ai/sdk` and `langfuse` as real dependencies (not present before this task) — first task that needs to call either. Langfuse tracing is implemented against the installed `langfuse@3.38.20` / `langfuse-core` API surface (`.trace()`/`.trace.update()`/`.flushAsync()`, confirmed against the package's own `.d.ts` rather than guessed) and gated the same way as the Anthropic key: no `LANGFUSE_SECRET_KEY`/`LANGFUSE_PUBLIC_KEY` configured → `startTrace` still returns a locally-generated UUID (`crypto.randomUUID()`) as `traceId` with a no-op `end`, so the `prompt_version` + trace id persistence contract holds even with tracing unconfigured.
- T24: template rendering for `prompts/premortem_v1.md` is a small hand-rolled mustache-lite (`{{var}}`, `{{#list}}...{{/list}}`) rather than a templating dependency — the prompt file only uses those two constructs, so no library pulled in for it.
- T23: AC says "desired defaults true" — that's the DB column default (`forecasts.desired default true`, T07) and the create form's checkbox is pre-checked (`defaultChecked={initial?.desired ?? true}`), so the ordinary UI path defaults to true. The server action itself, however, must read an *unchecked* HTML checkbox and an *absent* field identically (both omit the key from `FormData`), so `createForecast`/`updateForecast` compute `desired = false` whenever the field isn't present rather than defaulting to true at the action layer — the DB default only applies on a bare insert with no `desired` key at all, which the action never does (it always sets the field explicitly, one way or the other). Reused T22's decision-form pattern (list-add-remove, ownership+draft gate via service-role fetch, inline errors) for the forecast list/form rather than building a new pattern.

- T24 (operator recovery): the risk:high review found four correctness issues across cycles; the in-loop fixer cleared the transactional-orphan and TOCTOU races, then the escalation cap was exhausted with two low-severity findings left. An operator-driven fixer finished them — SDK transport errors now map to `{ok:false}` via a `callAndParse` try/catch (the action never rejects), and `renderTemplate` is single-pass so a forecast question containing `{{...}}` isn't re-expanded into the decision record. Fable re-review: pass. **Carry-forward for T26:** the TOCTOU window between the draft re-check and the premortem/risks inserts is narrowed, not closed — T26's commit action should verify no in-flight premortem (or add a DB guard). Also parking-lot: `startTrace` fabricates a uuid when Langfuse creds are absent (persisted as `langfuse_trace_id`); `null` would be more honest.
- CI / harness gap: `pnpm-workspace.yaml`'s `onlyBuiltDependencies` was empty, so pnpm 10+ errored (`ERR_PNPM_IGNORED_BUILDS`) on frozen install in CI even though the app built — approved `sharp`+`unrs-resolver`. The loop's local gate runs plain `pnpm check` against a warm `node_modules` and never runs `pnpm install --frozen-lockfile`, so a CI-only failure passes the gate. Worth closing that parity gap in the harness.
