# Proof-of-catch — T21 open redirect (Phase C)

**Date:** 2026-07-02 · **Reviewer:** `claude-fable-5`, read-only (`.claude/agents/reviewer.md`) · **Target:** commit `facda4e` from `origin/archive/harness-v1.1` — a loop-authored redirect "fix" that passed the old `pnpm check` gate green (see `docs/EVIDENCE.md`).

## What the author shipped

```ts
const next = rawNext.startsWith("/") && !rawNext.startsWith("//") ? rawNext : "/"
```

Looks like an open-redirect guard. Green build, tests passed, loop moved on.

## What the reviewer found (cold — no access to EVIDENCE.md, 0 tool uses)

`?next=/\evil.com` passes both checks — starts with `/`, does not start with `//` — so the raw value reaches `redirect("/\evil.com")`. Browsers fold `\` → `/` in the authority position, resolving it as the protocol-relative `//evil.com` → the freshly-authenticated user lands on `https://evil.com`. The reviewer enumerated the full vector space, confirmed the other four vectors (`//evil.com`, `https://evil.com`, `https:evil.com`, `http:/\evil.com`) are correctly rejected — a single-vector escape, which is what made it easy to miss — traced the one that escapes, and prescribed the origin-equality fix (`new URL(raw, origin)`, accept only if `url.origin === request.nextUrl.origin`), explicitly rejecting the whack-a-mole `!startsWith("/\\")` patch as incomplete. It also flagged the absent guard test.

**Verdict:** `VERDICT: flag | severity=high | class=security`
